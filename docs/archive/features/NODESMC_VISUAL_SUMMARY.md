## Key Design Decisions

### 1. Separate Methods for Split vs Non-Split
```python
# For Telegram (no splitting needed)
get_meshcore_paginated(page, days_filter)
  â†’ Returns: str

# For MeshCore (with splitting)
get_meshcore_paginated_split(page, days_filter, max_length)
  â†’ Returns: list[str]
```

**Why?**
- Different return types (str vs list[str])
- Telegram doesn't need splitting overhead
- MeshCore needs precise control over message boundaries
- Clearer separation of concerns

### 2. Line-Based Splitting
```python
# Split at line boundaries, not arbitrary positions
for line in lines:
    if current_length + line_length > max_length:
        # Start new message
    else:
        # Add to current message
```

**Why?**
- Preserves readability
- Never breaks in middle of node information
- Natural message boundaries
- Easy to understand output

### 3. Message Numbering
```python
# Only add numbering if multiple messages
if len(messages) > 1:
    for i, msg in enumerate(messages, 1):
        numbered.append(f"({i}/{len(messages)}) {msg}")
```

**Why?**
- User knows how many messages to expect
- Clear indication of message sequence
- Only adds overhead when needed (multi-part messages)

### 4. Congestion Control
```python
# 1-second delay between MeshCore messages
for i, msg in enumerate(messages):
    self.sender.send_single(msg, sender_id, sender_info)
    if i < len(messages) - 1:
        time.sleep(1)
```

**Why?**
- Prevents overwhelming mesh network
- Maintains message ordering
- Gives time for first message to be acknowledged
- Standard pattern used elsewhere in codebase
